### 2026-05-24 — fincli HTTP API umbrella patterns

**What:** Five durable patterns the `fincli_api/` rollout (branch `feat/fincli-api`, 9 commits ending at `e248ede`) established that future work in this repo should apply by default.

1. **Mock target for fincli HTTP in API tests is `fincli.app.main.fetch_page_sync`, NOT `fincli.utils.web_scraper.fetch_page_sync`.** The former is the local-name binding produced by `from fincli.utils.web_scraper import fetch_page_sync` at the top of `main.py`; patching the original location after `main.py` has already imported the name does not affect the local binding the orchestrator actually calls. T3 BACKEND discovered this empirically (mocked tests passed but live network calls still fired); T5b's conftest documents the rationale so the next agent does not re-learn it.

2. **Validator-first ordering for any non-interactive filter input.** `fincli/utils/quary_builders.py` lines 18-22 silently drop unknown filter keys with no error and no warning. Any route handler accepting structured filter input MUST call `validate_filter_pairs(tuple(filters.items()))` from `fincli/resource/params/validators.py` BEFORE invoking the screener adapter. Otherwise a typo'd key returns a deceptive 200/empty payload instead of the 422 the spec promises. The validator was carved out into its own module specifically to close this silent-drop hazard for non-interactive callers (interactive UI never produces invalid keys because it numbers them off a fixed menu).

3. **Single-source classifier extension over per-caller workarounds.** When the HTTP layer needed `click.UsageError` to map to USAGE/422, the correct fix was extending `fincli.app.exit_codes.classify()` (the single source of truth) to recognize `UsageError`. The wrong fix would have been a `try/except UsageError` in the route handler that bypassed `classify()`. That workaround would have left `classify()` silently pretending `UsageError` is unreachable — a latent footgun for the next caller. Extend the single source; do not fork the dispatch.

4. **`pytest.ini` is the canonical pytest config**, not `pyproject.toml`'s `[tool.pytest.ini_options]`. Pytest precedence picks `pytest.ini` first when both exist; T1's `pyproject` section was silently inert for the entire umbrella until T5c's `@pytest.mark.live` discovery exposed the drift. The fix consolidated all pytest config into `pytest.ini` and stripped `pyproject.toml`'s section to a single comment pointing at the canonical file. New marker registration, addopts, etc. go in `pytest.ini`.

5. **xfail-pair pattern for deferred bugs with future fix deadlines.** When MAJOR #4 (malformed HTML coerces to 200/empty instead of 502/UPSTREAM) was deferred from the umbrella scope, T5b paired a `@pytest.mark.xfail(strict=True)` test for the spec-intent shape with a current-behavior pin test that asserts the present (incorrect) status. Closing the deferred bug will mechanically trip BOTH tests in the same commit — the xfail becomes XPASS (strict=True hard-fails the suite) AND the pin test starts failing — forcing a coordinated code + test + docs update. The pattern is generalizable to any "we know it's wrong, we'll fix it later" gap.

**Why:** Each pattern was surfaced by VERIFIER or empirically discovered during BACKEND implementation across the 7-wave umbrella. Each saves ~30-60 minutes of debugging for the next agent who encounters the same gap. Pattern 3 (single-source classifier extension) is the deepest architectural rule — it is the "do not fork the spec when extending it" principle made concrete. Pattern 5 (xfail-pair) is the most reusable beyond this umbrella; any deferred bug that risks being forgotten until a downstream consumer trips over it benefits from the strict-xfail forcing function.

**How to apply:** When adding a new non-Python entry point to fincli (a Go consumer via subprocess, a webhook receiver, a CLI alias, a long-lived daemon), patterns 1-4 apply directly. Pattern 5 applies any time a bug fix needs to be deferred for a sizeable parser/library change — pair an `xfail(strict=True)` for the desired behavior with a current-behavior pin to force a coordinated PR later. The pipeline-mode umbrella's near-miss (1-page `IndexError` that shipped because mocked tests never exercised the live `page_count` path — captured in the 2026-05-22 entry below) is the durable rationale for the live-e2e tier (`pytest -m live tests/e2e/api/`) being mandatory before HUMAN approval on any change to `fincli_api/` or `fincli/stock_screening/`. Mocked tests catch regression; live tests catch the gaps the mocks lie about.

### 2026-05-22 — Screener pipeline hardening (page_count + OUTPUT_PATH= contract)

**What:** Fixed two pre-existing defects in the screener pipeline that live e2e against Finviz surfaced. (1) `fincli/stock_screening/content/stock_table.py:page_count` raised `IndexError` on responses with exactly 1 page of results (one `<a class="screener-pages is-selected">` element, no `is-next` arrow); the old `content[-2].string if len(content) != 0 else 0` had a 0-element guard but no 1-element guard. Replaced with a 0/1/≥2 three-branch dispatch using `.get_text(strip=True)`. (2) `fincli/app/main.py` left `resolved_file_path` unassigned on any pre-fetch exception path, so the `OUTPUT_PATH=` discovery line surfaced with empty value when `page_count` (or anything in the try-block) raised. Hoisted the `resolved_file_path = config.file_path("stock_screener")` assignment ABOVE the `try:` block — path resolution is pure (no I/O, no network) so it cannot itself raise the upstream/data exceptions the try-block is supposed to guard. Added one integration fixture (`tests/integration/fixtures/finviz_one_page.html`) + three regression tests (unit parametrized `page_count`, integration single-page CLI, unit `OUTPUT_PATH= populated on DATA exception`). Folded REVIEWER LOW doc-sync items: `CONTRACTS.md` §1/§5.3.3 + `docs/MODULE_REFERENCE.md:113` now explicitly state the `OUTPUT_PATH=` value is populated on every exit code (0/1/3/4). Type-hint NIT: added `-> int` return annotation to `page_count`.

**Why:** The list-filters umbrella's stacked PR (on top of refactor/fincli-only + Stop-hook cleanup) was about to ship, and the user asked the load-bearing question: "did you run a real world e2e test? running the project and checking that the outcome is what we want?" Answer was no — all umbrella validation went through `CliRunner` + mocked `fetch_page_sync` + canned 354-byte `finviz_happy.html` fixture, which has *zero* `screener-pages` elements and falls through the `len != 0 else 0` guard. The mocked test surface never represented Finviz's actual 1-element pagination shape. Shipping the umbrella on top of a screener that crashed on every 1-page filter combination would have negated the umbrella's premise (fincli is consumable by a downstream Go pipeline). The `OUTPUT_PATH=` empty-value bug was a contract violation of pipeline-mode-spec §5.3.3, hidden until the parser crash exposed it.

**How to apply:** Three durable rules, all earned by this incident. (1) **Mocked tests are necessary but not sufficient for a CLI's primary use case.** The pipeline-mode umbrella shipped ~200 unit/integration tests, all passing — and missed a 1-page Finviz bug that fires on the most basic filter combo. New CLI features that touch the screener pipeline should include at least one live-Finviz smoke test invocation in the validation gate before HUMAN approval, not just CliRunner mocks. The user's explicit "did you run a real world e2e test?" prompt should be the default question, not a save. (2) **Synthetic fixtures must mirror the realistic narrow-result shape, not just the happy multi-page case.** Future BS4 parser fixtures should include a 1-page sibling alongside the multi-page fixture (`finviz_happy.html` → `finviz_one_page.html` pattern established here). The Phase-2 deferred work on real parser tests has this as one of its deliverables. (3) **Spec invariants like `OUTPUT_PATH=` non-empty must be enforced by the unit suite, not just documented.** The new `test_output_path_populated_on_data_exception` is the kind of test that would have caught this on the original Pillar-4 task; future spec invariants should ship with a paired enforcement test that exercises the failure path, not just the success path.

### 2026-05-21 — Filter inventory dump (`--list-filters --json`) + INTEGRATION.md

**What:** Shipped the `list-filters-spec.md` feature over three tasks landing in commits `82d3d6e` (T1: helpers), `6a8c343` (T2: CLI wiring), and the umbrella-closer T3 (this doc sweep + spec archive). `fincli --list-filters --json` dumps the full Finviz filter inventory as a single-line JSON object (`{schema_version, keys, filters: {key: {label, values}}}`) to stdout and exits 0, short-circuiting the screener pipeline. A new top-level `INTEGRATION.md` documents the subprocess pattern for non-Python consumers (Go, Node, Rust). The spec moved to `docs/features/archive/list-filters-spec.md` with a SHIPPED banner. Earlier deep-think pass at `16c79ec` amended both the spec and the plan with the canonical `keys` ordering contract before T1 began.

**Decisions captured:**

- **Polyglot framing**: JSON is the format; all four shape options considered were JSON. Picked the nested-with-labels shape (`{label, values: {...}}`) over flat codes-only or grouped-by-category. Anti-polyglot Python-tuple form was rejected upfront. Spec §5.2.

- **Mechanical label derivation, not data augmentation.** `fincli/resource/params/_label_format.attr_to_label` derives display labels from Python attribute names (e.g., `FORWARD_PE` -> `"Forward PE"`) with acronym preservation (`PE`, `ROA`, `EPS`, `RSI`, ...) and connector lowercasing (`to`, `and`, `of`, ...). Avoids touching the params files' two-element-list contract (CONTRACTS §2). Known cosmetic limitations (preserves the `Fifty_Tow_Day_High_Low` typo; `PE` renders as `PE` not `P/E`; `QTR` renders as `Qtr`) are intentional starting-point trade-offs that consumers can override locally. Spec §5.3 + OQ4/OQ5.

- **`LIST_FILTERS_SCHEMA_VERSION = 1` pinning**, mirroring the `JSON_SUMMARY_SCHEMA_VERSION` pattern from pipeline mode (CONTRACTS §5.5). Adding new filter entries or new value codes is **additive** and does NOT bump the version; rename/remove/type changes bump it and are breaking. The constant is a module-public name (no leading underscore) so polyglot integrators with a Python shim can `from fincli.app.cli import LIST_FILTERS_SCHEMA_VERSION` rather than re-typing the literal.

- **`keys` array as the canonical ordering contract** (deep-think amendment at `16c79ec`). Go's `encoding/json` decode into `map[string]T` randomizes iteration; JS object iteration is engine-defined. The top-level `keys` list pins the Fundamental -> Descriptive -> Technical declaration order so consumers can iterate it and index into `filters[key]`. `set(keys) == set(filters.keys())` and `len(keys) == len(filters)` are tested invariants. The key-ordering nuance — `sh_*` prefix appears in both `Fundamental_Params` (ownership) and `Descriptive_Params` (shares/volume) but class membership wins over prefix grouping — is documented in spec §5.2 and CONTRACTS §5.6.

- **`--list-filters`, not `--help-filters`.** Leaves `--help-filters` available for a future human-readable terminal view (currently deferred). Spec §3 / §9.

- **No HTTP server / no `fincli serve`.** Subprocess pattern stays the integration boundary — the pipeline-mode umbrella made the CLI deterministic enough to be a single-shot building block, and INTEGRATION.md documents how to use it as one. Spec §3 / §9.

- **Per-language cookbook deferred.** INTEGRATION.md lands the language-agnostic patterns now (bootstrap, per-screen call flow, exit-code routing table, `OUTPUT_PATH=` recovery, concurrency notes, caching guidance); working Go / Node / Rust examples are placeholder-only until the downstream consumer project is further along. Spec §9 + INTEGRATION.md "Per-language cookbook" section.

- **`--json` flag is silently ignored when `--list-filters` is not set** (spec OQ2 HUMAN-approved default). Sub-flag of `--list-filters`, not a free-standing format selector — adding bare `--json` to an unrelated invocation must not break it. Pinned by an explicit unit test (`tests/unit/app/test_cli_list_filters.py::test_bare_json_flag_is_silently_ignored`) per spec OQ-E.

- **Integrated OQ-B/C/D matrix test** (BACKEND step 9, per the `16c79ec` deep-think follow-up). One matrix-style test pins the short-circuit + mutex + orthogonal-flag interaction as a single unit: `--list-filters --json --quiet --debug --json-summary --output -` exits 0, stdout is a single JSON line, no banner anywhere, no progress logs, no `OUTPUT_PATH=` line, no JSON summary. Pins the **interaction** (short-circuit fires BEFORE banner emission, BEFORE the `run_stock_screener` import, BEFORE each orthogonal-flag handler), not isolated rules.

**Not shipped (tracked):**

- Per-language cookbook code examples (Go, Node, Rust). Placeholder in INTEGRATION.md awaits the downstream consumer project.
- `--help-filters` (human-readable terminal variant). Reserved name; could be added later.
- Other format flags (`--list-filters --yaml`, `--list-filters --text`). `--json` is the only currently-supported format.
- Typo fix for `Fifty_Tow_Day_High_Low` (-> `Two`). Separate one-line PR; this spec preserved the existing label to avoid conflating cleanup with feature work.

**How to apply:** Existing dev environments need no reinstall — no new runtime dependency. The new flags layer onto the existing CLI with zero behavioral change for users who don't set them. For non-Python integrators: read `INTEGRATION.md` for the subprocess pattern; CONTRACTS §5.6 for the inventory JSON schema; CONTRACTS §5.5 for the per-run summary schema; `fincli/app/exit_codes.py` for the classifier constants. For Python integrators: continue to use the importable surface in `core.configuration.configurator.build_config` + `fincli.app.main.run_stock_screener` per CONTRACTS §6.

### 2026-05-16 — Pipeline mode (structured input + deterministic output + stream discipline + exit codes)

**What:** Shipped the `pipeline-mode-spec.md` umbrella feature over six tasks landing across commits `f775b7e..HEAD` on the `refactor/fincli-only` branch. fincli is now usable as a single-shot building block in a downstream automation pipeline. The spec moved to `docs/features/archive/pipeline-mode-spec.md` with a Shipped banner.

**Decisions captured:**

- **Canonical filter-input shape: flat JSON object** `{"fa_pe":"u20",...}`. List shape no longer accepted. `core/converters/json.py` raises `ValueError` on any non-canonical shape; CLI translates to `click.UsageError` (exit 2). Framed in CONTRACTS §6.3 as a **conformance fix**, not a SemVer-style break — the legacy list-shape path silently fed unvalidated pairs into `quary_builders.build_stock_screener_query`, which then silently dropped unknowns (the exact silent-corruption failure mode THESIS Principle #2 prohibits).
- **Output destination precedence:** `--output PATH` > `--output -` > `FINCLI_OUTPUT_DIR` env > `Config.file_path()` default. Pinned at one site in `Config.file_path` so the CLI / orchestrator never branch on precedence separately.
- **Stream routing keyed on `--output -`,** not on a TTY check. Spec refinement E2 — TTY autodetection produces surprises under `nohup`, `>` redirection, CI runners, Windows job objects.
- **`--quiet` and `--debug` are orthogonal axes.** `--debug` wins on logger level; `--quiet` wins on console suppression. File handlers (`activity.log` / `error.log`) are unaffected by `--quiet`.
- **Five exit codes** (0 SUCCESS / 1 INTERNAL / 2 USAGE / 3 UPSTREAM / 4 DATA). Click owns USAGE; the orchestrator owns the other four via `fincli.app.exit_codes.classify`. Tests import the named constants so a future renumbering touches one file.
- **`Symbol` is the canonical machine-readable ticker column.** `Ticker` keeps the `=HYPERLINK(...)` Excel wrap **except under `--output -`** where it's the raw symbol so `pandas.read_csv` consumers downstream aren't poisoned by formulas. Column order unchanged (regression-pinned).
- **Zero-row results write a header-only CSV.** Previously a silent-return left `OUTPUT_PATH=` (empty value) on stderr — a contract violation since exit was 0 but no CSV existed. The header-only write resolves the destination, writes the file (or streams the header line), and emits the populated `OUTPUT_PATH=<abs>` discovery line.
- **`OUTPUT_PATH=<value>` stderr discovery line** is unconditional — emitted on every run regardless of `--quiet` / `--json-summary` / success or failure. Pipeline integrators can recover the destination via `tail -n1 stderr | cut -d= -f2-`. Now governed by CONTRACTS §7 stability policy.
- **`--json-summary` schema is version-pinned** (`schema_version: 1`). Adding fields is non-breaking; removing or renaming bumps the version. Now governed by CONTRACTS §7.
- **`Market Cap` dtype tightened to nullable `Float64`.** Reframed in CONTRACTS §3.1 as a conformance fix (was declared `float` but the legacy code returned `float | str`). Landed in commit `50f46ca`. The `convert_market_cap_to_numeric` function moved out of `fincli/app/main.py` into its own module `fincli/utils/market_cap.py` for direct testability.
- **`json_to_tuples` schema lockdown** framed as a conformance fix matching the Market Cap precedent (tightening implementation toward documented contract is a fix, not a redefinition). The legacy list-shape acceptance was silent-corruption fuel.
- **`filter_history.json` writeback** moved out of the `if config.use_history:` block in `select_filters_and_values` (pre-existing bug — the gate was the read path, not the write path). Now triggers on every successful run that produced a non-empty filter set, regardless of input mode.

**Not shipped (tracked):**

- Default output-path portability (`workspace_output/` under CWD remains the default). Tracked in `docs/features/archive/pipeline-mode-spec.md` §9 — needs its own migration story and FEEDBACK-LOG entry.
- `--fail-on-empty` opt-in flag. Genuine pipeline use cases want zero-row = success (today's Finviz universe + tight filters legitimately produces empty). Future feature spec.
- `HISTORY_DIR` → `FINCLI_HISTORY_DIR` rename. Backward-compatibility window needed; cleaner as its own focused spec.
- `--help-filters` CLI flag. Helper function lands but the CLI flag does not (avoids decision creep). Trivial one-Click-option follow-up.

**How to apply:** Existing dev environments need no reinstall — every dependency was already on the requirements list. The new options layer onto the existing interactive flow with zero behavioral change for users who don't set them. For pipeline integrators: read the new "Pipeline mode" section in `README.md` for cookbook examples; CONTRACTS §1 has the full option table + behavior matrix + exit-code table; the JSON summary schema is CONTRACTS §5.5. The classifier constants live in `fincli/app/exit_codes.py` and should be imported by hand (not hardcoded) by any test or downstream tool that branches on exit codes.

### 2026-05-11 — Restore `--scrape-link` CLI option

**What:** Re-added the `--scrape-link=<url>` Click option to `fincli/app/cli.py:run_main`. The option threads through `run_stock_screener(scrape_link=...)` → `build_config(scrape_link=...)` → `Config.scrape_link`. When `Config.scrape_link` is truthy, `run_stock_screener` short-circuits both the interactive filter picker and the Finviz query builder, using the supplied URL verbatim as the screener query. `--scrape-link` and `--history` are mutually exclusive — combining them raises a Click `UsageError`. CONTRACTS §1 re-documents the option in the Options table, the Behavior table (incl. the mutual-exclusion row), and the §6.1/§6.2 signatures. The first regression test for the project landed at `tests/unit/app/test_cli.py` (two `CliRunner` tests: option-accepted, mutual-exclusion). No URL validation is performed; cfscrape surfaces invalid-URL failures downstream. The shipped spec moved to `docs/features/archive/scrape-link-restoration-spec.md` (creating `docs/features/archive/`).

**Why:** The option existed historically in the now-deleted `fundainsight/` package (`@click.option('--scrape-link', default="", ...)` on `fundainsight/app/cli.py`). Commit `a840a1c` ("refactor: remove fundainsight package and yahooquery dependency", 2026-05-05) deleted the package and the decorator with it. The fincli-side `fincli/app/cli.py` never carried `--scrape-link`, so this is feature restoration on a different entry point, not a true revert. Evidence the loss was incidental rather than deliberate: `Config.scrape_link: str = ""` field still exists; CONTRACTS §4, ARCHITECTURE, README, and `docs/MODULE_REFERENCE.md` all describe the field; the May-4 reduction spec and FEEDBACK-LOG say nothing about removing the feature; no commit removed `Config.scrape_link` deliberately. The bypass-the-interactive-flow capability is a major productivity feature for the sole user, especially for re-running a saved Finviz URL.

**How to apply:** Existing dev environments need no reinstall — Click was already a runtime dependency and `CliRunner` is built into Click. Run the screener with `fincli --scrape-link=<finviz-url>` (or `python -m fincli --scrape-link=<url>`) to bypass the interactive picker. The sibling `--set-filters` JSON-input option (also lost in `a840a1c`) was explicitly deferred from this cycle per user direction; if needed, it can be restored separately following the same pattern. The new test file is the first real test body in the repo and seeds the Phase 2 testing direction at the CLI option-parsing layer; `tests/unit/app/test_cli.py` is collected and passes under the existing `pytest.ini_options` config without any new infrastructure.

### 2026-06-27 — Removed the `run.sh` / `run.bat` convenience launchers

**What:** Deleted `run.sh`, `run.bat`, and `scripts/check_requirements.py`. Live docs updated to point at the canonical entry points only: `pip install -e ".[dev]"` then `fincli` / `python -m fincli` (CLI) and `uvicorn fincli_api.main:app` / `fincli-api` (HTTP API). References scrubbed from `README.md`, `CLAUDE.md`, `CONTRACTS.md` §1.1, `ARCHITECTURE.md`, `TOOLS_REFERENCE.md`, `agents/rules/scaffold-module.md`, `agents/roles/frontend-developer.md`, and `docs/THESIS.md`. Historical/archive docs (specs, handoffs, prior FEEDBACK-LOG entries, the arch-conversation transcript) retain their mentions as immutable record.

**Why:** The launchers' requirements-bootstrap had rotted into a broken no-op: `scripts/check_requirements.py` imported `pkg_resources`, which is absent in the current venv (`ModuleNotFoundError`), so the dep-check exited non-zero on a traceback every run — silently triggering `pip install -r requirements.txt` on every launch. And `requirements.txt` itself was stale (missing `beautifulsoup4`, `colorama`, `pydantic`, `pydantic-settings`, `fastapi`, `uvicorn`), so bootstrapping from it produced a non-working env. No consumer depended on the launchers — the orchestrator's `algo_beta` adapter calls `fincli` / `python -m fincli` directly; the API runs via uvicorn. `run.bat` additionally carried a `pause` (hangs non-interactive callers) and the `cmd.exe` `&`-truncation footgun that started this session: a Finviz `--scrape-link` URL invoked through the `.bat` from PowerShell was silently truncated at the first `&`, dropping all filters.

**How to apply:** There is no shell-launcher path anymore — install once with `pip install -e ".[dev]"`, then invoke `fincli` (or `python -m fincli` when the venv `Scripts/` dir is not on PATH). On Windows/PowerShell, quote `&`-bearing URLs and call the module directly — `python -m fincli "--scrape-link=…&f=…"` — or use the `--%` stop-parsing token; do not route `&`-bearing args through a `.bat`. New modules expose runnable CLIs via `pyproject.toml [project.scripts]`, not by re-adding launcher scripts. `requirements.txt` is retained (still used by `pip-audit -r requirements.txt` and the `urllib3<2` pin) but is not an install path for the app.
