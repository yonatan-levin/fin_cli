# Fin CLI HTTP API — Design Specification

**Version:** 0.1 DRAFT
**Date:** 2026-05-22
**Status:** DESIGN
**Author:** yonatan
**Mode:** PLAN_AND_CREATE (new module + cross-cutting docs sweep)
**Builds on:** Pipeline-mode + list-filters umbrellas (`docs/features/archive/pipeline-mode-spec.md`, `docs/features/archive/list-filters-spec.md`), `INTEGRATION.md`

---

## 1. Summary

Add a second co-equal entry point to the `fincli` package: an HTTP API server (`fincli_api/`) exposing the existing screener and filter-inventory capabilities over REST+JSON. The API is a thin layer over `fincli.app.main.run_stock_screener` and `fincli.resource.params.validators.list_valid_filters_with_labels` — same Python ecosystem, same orchestrator, same fetch+parse pipeline, no subprocess overhead. The HTTP contract is the primary artifact (formal OpenAPI 3.0 spec, auto-generated by FastAPI from typed endpoints, importable into Postman with one click); the Python implementation is the reference server. The project's identity shifts from "Fin CLI" to "Fin CLI + API" — two co-equal entry points to the same underlying screener logic.

Initial scope is intentionally minimal: single-user personal use via Postman against localhost, two endpoints (`GET /filters`, `POST /screens`), no auth, no persistence, no rate limiting. Auth / multi-tenancy / persistence become separate specs when a second caller materializes.

## 2. Goals and Non-Goals

### Goals

- **G1** A new sibling package `fincli_api/` lives next to `fincli/` in the same repo. Its only fincli dependency is the import boundary `fincli_api/adapters/fincli.py`.
- **G2** Two REST endpoints work end-to-end against live Finviz:
  - `GET /filters` returns the same `{schema_version, keys, filters}` payload as `fincli --list-filters --json`, byte-equivalent modulo HTTP framing.
  - `POST /screens` accepts `{"filters": {<key>: <value>, ...}}` and returns a `ScreenResult` with stocks, row count, and per-run metadata.
- **G3** FastAPI auto-generates a valid OpenAPI 3.0 spec served at `/openapi.json`. Swagger UI at `/docs`, Redoc at `/redoc`. Postman imports `/openapi.json` directly.
- **G4** All fincli exit codes (0/1/2/3/4) map cleanly to HTTP statuses (200/500/422/502/502) via the existing `fincli.app.exit_codes.classify` function. Reuse, don't reinvent.
- **G5** Three-tier test pyramid:
  - Unit (`tests/unit/api/`) — FastAPI TestClient + mocked adapter. <500ms total.
  - Integration (`tests/integration/api/`) — TestClient + real adapter + mocked `fetch_page_sync`. Reuses existing HTML fixtures.
  - E2E (`tests/e2e/api/`) — TestClient + real fincli + live Finviz. `@pytest.mark.live`, opt-in via `pytest -m live`.
- **G6** A second `[project.scripts]` entry — `fincli-api = "fincli_api.main:main"` — runs uvicorn. Symmetric with existing `fincli` entry point.
- **G7** All cross-cutting docs (`CLAUDE.md`, `THESIS.md`, `CONTRACTS.md`, `INTEGRATION.md`, `MODULE_REFERENCE.md`, `README.md`, `TESTING.md`, `FEEDBACK-LOG.md`) reflect the two-entry-point identity. New `docs/api/openapi.yaml` committed snapshot for change tracking.
- **G8** Live-Finviz e2e test (the `pytest -m live` gate) is enforced as a pre-PR check for any change to `fincli_api/` OR to `fincli/stock_screening/`. The list-filters umbrella's near-miss is the durable reason — captured in `FEEDBACK-LOG.md`.

### Non-Goals

- **N1** Authentication / authorization. Localhost-only personal use; if a second caller materializes, that's a separate spec.
- **N2** Rate limiting (per-caller, per-IP, or global). Same rationale as N1.
- **N3** Persistence (DB-backed screen history, saved filter sets, user accounts). Same rationale.
- **N4** Streaming responses (Server-Sent Events for long-running screens). Personal-scale screens complete in <10s; sync responses suffice.
- **N5** API versioning prefix (`/v1/...`). Premature for personal use; added when the first breaking change forces it.
- **N6** Subprocess-per-request shell-out to `python -m fincli`. Direct import is strictly simpler and faster; the polyglot subprocess contract continues to be served by the CLI entry point per `INTEGRATION.md`.
- **N7** A frontend (web UI, native app, TUI). Postman is the client. Frontends are a separate concern.
- **N8** The "detailed analysis features" the user described as TBD. Explicitly out of scope; a future separate spec when the feature set is defined.
- **N9** Concurrent fincli execution / worker pools. Single-user load; sync FastAPI endpoint functions running in FastAPI's default thread pool suffice.
- **N10** Production deployment hardening (TLS, observability, structured logging integration, container packaging, healthcheck beyond `/healthz`).

## 3. Architecture

### 3.1 Module layout

```
fin_cli/
├── fincli/                       # existing CLI (unchanged)
├── fincli_api/                   # NEW sibling package
│   ├── __init__.py
│   ├── main.py                   # FastAPI app, router wiring, uvicorn main()
│   ├── routes/
│   │   ├── __init__.py
│   │   ├── filters.py            # GET /filters
│   │   ├── screens.py            # POST /screens
│   │   └── meta.py               # GET /healthz
│   ├── models/
│   │   ├── __init__.py
│   │   ├── filters.py            # Pydantic — FilterInventory, FilterEntry
│   │   ├── screens.py            # Pydantic — ScreenRequest, ScreenResult, Stock
│   │   └── errors.py             # Pydantic — ErrorResponse + discriminator union
│   ├── adapters/
│   │   ├── __init__.py
│   │   └── fincli.py             # only file that imports from fincli/
│   ├── exception_handlers.py     # @app.exception_handler routes via classify()
│   └── config.py                 # ApiConfig (host, port, log level — Pydantic Settings)
└── tests/
    ├── unit/api/                 # FastAPI TestClient + mocked adapter
    ├── integration/api/          # TestClient + real fincli + mocked Finviz
    └── e2e/api/                  # TestClient + real fincli + live Finviz
```

### 3.2 Boundary — `fincli_api/adapters/fincli.py`

The only file in `fincli_api/` that imports from `fincli/`. Two functions:

```python
def get_filter_inventory() -> dict:
    """Wraps fincli.resource.params.validators.list_valid_filters_with_labels.
       Returns the same {schema_version, keys, filters} shape the CLI's
       --list-filters --json emits (single source of truth — both paths
       call the same Python function)."""

def run_screen(filters: dict[str, str]) -> ScreenResult:
    """Calls fincli's screener orchestrator with structured filter input;
       returns parsed stock rows + run metadata. Never writes a CSV file;
       never spawns a subprocess. Reuses the URL-builder fincli already
       uses for the Excel HYPERLINK wrap so `finviz_url` cannot drift."""
```

To return a DataFrame in-memory rather than writing CSV, `fincli/app/main.py` gains a one-time small helper (`screen_to_dataframe(filters, ...) -> pd.DataFrame` or similar) that the existing `run_stock_screener` also calls internally for the CSV write path. **This is a non-breaking addition to fincli**, not a refactor of the existing CLI flow.

### 3.3 Process model

- **Sync FastAPI endpoint functions** (`def`, not `async def`). fincli is fully sync (cfscrape + BS4 + pandas). FastAPI runs sync endpoints in its default thread pool — fine for I/O-bound HTTP scraping at single-user load.
- **No subprocess** — direct Python imports throughout.
- **Single process per dev session** — `uvicorn fincli_api.main:app --reload` or `python -m fincli_api`. No worker pool, no auto-restart, no daemonization.

### 3.4 Deployment posture

Localhost only. Default `--host 127.0.0.1 --port 8000`. No TLS, no auth, no CORS, no rate limits. Any change to this posture is gated on a future spec.

## 4. API Surface

### 4.1 Endpoint table

| Method | Path | Purpose | Request body | Response body |
|---|---|---|---|---|
| `GET` | `/filters` | Dump the full filter inventory | (none) | `FilterInventory` |
| `POST` | `/screens` | Run a screen with provided filters | `ScreenRequest` | `ScreenResult` (200) / `ErrorResponse` (4xx/5xx) |
| `GET` | `/healthz` | Liveness check | (none) | `{"status": "ok"}` |
| `GET` | `/openapi.json` | OpenAPI 3.0 spec (FastAPI auto-generated) | (none) | OpenAPI dict |
| `GET` | `/docs` | Swagger UI (FastAPI auto-generated) | (none) | HTML |
| `GET` | `/redoc` | Redoc UI (FastAPI auto-generated) | (none) | HTML |

No version prefix; see N5.

### 4.2 Request — `POST /screens`

```python
class ScreenRequest(BaseModel):
    filters: dict[str, str] = Field(
        ...,
        examples=[{"fa_pe": "u5", "sec": "energy"}],
        description="Map of Finviz filter key → value code. See GET /filters for the valid set.",
    )
```

```json
{"filters": {"fa_pe": "u5", "sec": "energy"}}
```

Wrapping `filters` under a top-level key leaves room for future `options`/`pagination`/etc. without breaking the schema.

### 4.3 Response — `POST /screens` (200 OK)

```python
class ScreenResult(BaseModel):
    schema_version: int = 1
    row_count: int
    duration_ms: int
    started_at: str        # ISO 8601
    finished_at: str
    stocks: list[Stock]

class Stock(BaseModel):
    ticker: str
    company: str
    sector: str
    industry: str
    country: str
    market_cap: float | None        # nullable per fincli/utils/market_cap.py
    pe: str | None                  # "n/a" possible; preserved as fincli formats it
    price: str                      # "$25.41" — preserved
    change: str                     # "+1.23%" — preserved
    volume: str                     # "1.2M" — preserved
    rank: int                       # mapped from Finviz "No." column (1-based)
    finviz_url: str                 # https://finviz.com/quote.ashx?t={ticker}
```

**Field-naming rule:** snake_case for all fields. Verbatim Finviz column names (`Ticker`, `Market Cap`, `P/E`, `No.`) are normalized to snake_case in the API contract (`ticker`, `market_cap`, `pe`, `rank`). The OpenAPI spec documents the mapping.

**Numeric vs string preservation rule:** numeric for what's already numeric in fincli's coercion (`market_cap`), string for everything Finviz formats with units/currency/percent (`pe`, `price`, `change`, `volume`). Consumers opt in to numeric parsing.

```json
{
  "schema_version": 1,
  "row_count": 2,
  "duration_ms": 1843,
  "started_at": "2026-05-22T15:23:01.234Z",
  "finished_at": "2026-05-22T15:23:03.077Z",
  "stocks": [
    {"ticker": "CNX", "company": "CNX Resources Corporation", "sector": "Energy",
     "industry": "Oil & Gas E&P", "country": "USA", "market_cap": 5234000000.0,
     "pe": "4.2", "price": "$34.55", "change": "+1.23%", "volume": "1.2M", "rank": 1,
     "finviz_url": "https://finviz.com/quote.ashx?t=CNX"}
  ]
}
```

### 4.4 Response — `GET /filters` (200 OK)

```python
class FilterEntry(BaseModel):
    label: str
    values: dict[str, str]              # value_code -> human label

class FilterInventory(BaseModel):
    schema_version: int = 1
    keys: list[str]                     # canonical ordering, same as --list-filters --json
    filters: dict[str, FilterEntry]
```

Byte-equivalent to `fincli --list-filters --json` (modulo HTTP framing). Both paths call `list_valid_filters_with_labels` — they cannot drift.

## 5. Error handling

### 5.1 fincli exit code → HTTP status mapping

| fincli exit | Meaning | HTTP status | Response model | `error_class` |
|---|---|---|---|---|
| 0 (SUCCESS) | Screen ran, returned ≥0 stocks | `200 OK` | `ScreenResult` | (n/a) |
| 2 (USAGE) | Unknown filter key/value | `422 Unprocessable Entity` | `ErrorResponse` | `validation` |
| 3 (UPSTREAM) | Finviz down/timeout/blocked | `502 Bad Gateway` | `ErrorResponse` | `upstream` |
| 4 (DATA) | Finviz returned malformed HTML | `502 Bad Gateway` | `ErrorResponse` | `parsing` |
| 1 (INTERNAL) | Unclassified bug in fincli | `500 Internal Server Error` | `ErrorResponse` | `internal` |

**Why 502 for both 3 and 4:** from the API's perspective, both mean "the upstream gave us something we couldn't work with." The `error_class` field discriminates.

**Why 422 (not 400) for validation:** 422 is more precise for syntactically-valid JSON with semantically-invalid filter key (what `validate_filter_pairs` raises `click.UsageError` for). 400 is reserved for FastAPI's automatic handling of malformed JSON / missing required fields.

### 5.2 Error envelope

```python
class ErrorResponse(BaseModel):
    schema_version: int = 1
    error_class: Literal["validation", "upstream", "parsing", "internal"]
    message: str                           # human-readable, safe to surface in UI
    details: dict | None = None            # error-class-specific structured payload
    request_id: str | None = None          # for cross-referencing with logs (5xx only)
```

Examples by class:

**`validation` (422)** — unknown filter key:
```json
{"schema_version": 1, "error_class": "validation",
 "message": "Unknown filter key 'fa_pee'. Did you mean 'fa_pe'?",
 "details": {"key": "fa_pee", "suggestions": ["fa_pe", "fa_peg", "fa_fpe"]}}
```

**`upstream` (502)** — Finviz timeout:
```json
{"schema_version": 1, "error_class": "upstream",
 "message": "Finviz request timed out after 10 seconds.",
 "details": {"url": "https://finviz.com/screener.ashx?...", "timeout_s": 10}}
```

**`parsing` (502)** — BS4 IndexError or AttributeError:
```json
{"schema_version": 1, "error_class": "parsing",
 "message": "Failed to parse Finviz response — Finviz HTML structure may have changed.",
 "details": {"exception_type": "IndexError", "exception_message": "list index out of range"}}
```

**`internal` (500)** — unclassified bug:
```json
{"schema_version": 1, "error_class": "internal",
 "message": "An internal error occurred. Please report this with the request_id.",
 "request_id": "req_abc123"}
```

### 5.3 Implementation pattern

`fincli_api/exception_handlers.py` registers a single `@app.exception_handler(Exception)` that:

1. Calls `fincli.app.exit_codes.classify(exc)` — the existing classifier is the single source of truth for "what kind of failure was this."
2. Maps the returned integer to HTTP status (table in §5.1).
3. Wraps in the appropriate `ErrorResponse` shape with `error_class`.
4. Logs the full traceback (separate from response body).
5. Generates a UUID4 `request_id` for 5xx paths.

FastAPI handles `400 / 404 / 405 / 422-from-Pydantic` automatically; our handler only takes the 500-catch-all and re-shapes it via the classifier.

## 6. Testing strategy

Three tiers; numbers approximate.

### 6.1 Unit — `tests/unit/api/` (~12-15 tests, <500ms total)

FastAPI `TestClient` against route handlers with `adapters.fincli` mocked. Covers every route, every error-class branch, OpenAPI generation, `/healthz`, plus FastAPI's automatic 422/404 behavior.

### 6.2 Integration — `tests/integration/api/` (~8-10 tests, <3s total)

`TestClient` against real adapter + real fincli orchestrator + mocked `fincli.utils.web_scraper.fetch_page_sync`. Reuses the five HTML fixtures the umbrella + screener fix already shipped (loaded via `tests/integration/_fixtures_loader.py`):

- `finviz_happy.html` + `finviz_happy_html()` — multi-page success
- `finviz_one_page.html` + `finviz_one_page_html()` — 1-page success (from screener-fix commit `8d22af7`)
- `finviz_empty.html` + `finviz_empty_html()` — zero-row success
- `finviz_no_table.html` + `finviz_no_table_html()` — Finviz returned non-table HTML (parsing error path)
- `finviz_malformed_row.html` + `finviz_malformed_row_html()` — Finviz HTML with broken row (parsing error path)

Covers: happy path with each success fixture, zero-row case (200 with empty `stocks` list and `row_count: 0`), single-page case (the bug that the umbrella's mocked tests missed), each exit-code → HTTP-status branch via real exception propagation through the two parsing-error fixtures.

### 6.3 E2E — `tests/e2e/api/` (~3 tests, <30s total, opt-in)

`TestClient` against real fincli + live Finviz. Marked `@pytest.mark.live`; excluded by default (`pytest -m "not live"` in `pyproject.toml` `addopts`). Run via `pytest -m live tests/e2e/api/`.

- `GET /filters` returns 66 keys
- `POST /screens` with `fa_pe=u5 + sec=energy` returns ≥1 stock
- `POST /screens` with `fa_pe=u5 + sec=basicmaterials + geo=monaco` returns 0 stocks at 200

This is the gate that catches the kind of drift the umbrella's mocked tests missed. **It is mandatory before HUMAN approval on any change to `fincli_api/` or `fincli/stock_screening/`.**

### 6.4 Coverage philosophy

- New `fincli_api/` code: ≥90% line coverage from day 1 (FastAPI TestClient makes this cheap)
- No backfill of fincli's coverage — still Phase-2-deferred for the project
- Integration tier IS the coverage-by-other-means for fincli's adapter surface

## 8. Acceptance criteria

A reviewer verifies these in <10 minutes:

1. `uvicorn fincli_api.main:app` starts on port 8000 with no errors.
2. `curl http://localhost:8000/healthz` → `{"status": "ok"}`, HTTP 200.
3. `curl http://localhost:8000/filters` → valid JSON matching `--list-filters --json` byte-equivalently.
4. `curl -X POST http://localhost:8000/screens -H "Content-Type: application/json" -d '{"filters": {"fa_pe": "u5", "sec": "energy"}}'` → 200 with `ScreenResult` shape; `stocks[*].finviz_url` present and well-formed.
5. `curl -X POST .../screens -d '{"filters": {"fa_pee": "u5"}}'` → 422 with `error_class: "validation"`.
6. `http://localhost:8000/docs` shows Swagger UI with both endpoints + all schemas.
7. `http://localhost:8000/openapi.json` returns valid OpenAPI 3.0; loadable into Postman via "Import OpenAPI URL".
8. `pytest tests/` runs full unit + integration suite green (default `-m "not live"` excludes E2E).
9. `pytest -m live tests/e2e/api/` runs the three live-Finviz smoke tests green.
10. `ruff check . && ruff format --check . && mypy fincli fincli_api` all clean.

## 9. Scope guardrails

In scope:
- Two endpoints with the exact shapes in §4
- OpenAPI auto-generation + Swagger UI + Redoc
- Three-tier test pyramid (§6)
- Doc sweep across the files in §7
- One-time small helper in `fincli/app/main.py` to return a DataFrame in-memory

Explicitly out of scope (each becomes a future separate spec if needed):
- Authentication, rate limiting, persistence, observability, deployment hardening (`N1-N3, N10`)
- API versioning prefix `/v1/...` (`N5`)
- Streaming responses, gRPC, GraphQL (`N4`)
- Frontends (web/native/TUI) (`N7`)
- The user's TBD "detailed analysis features" (`N8`)
- Subprocess fallback to `python -m fincli` (`N6`)
- Worker pools / concurrency tuning (`N9`)
