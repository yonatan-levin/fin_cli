SECTION 6 — Handoff notes for the BACKEND implementer
------------------------------------------------------------------------------

These notes are reasoning the spec doesn't carry — narrative guidance for
the implementer that complements the formal contract.

1. THE ONE-LINER IS THE WHOLE GAME, FOR ORDERING PURPOSES.

   Steps 1, 4, 5, 6, 7, 8, 9 of the commit ordering can be reordered among
   themselves with no functional consequence. Steps 2 and 3 cannot. The
   reason is concrete: between the deletion of fundainsight/ and the
   configurator fix, `python -m fincli --history` would attempt to read
   `os.path.realpath('fundainsight')/local_history/filter_history.json` —
   a path that no longer exists — and raise FileNotFoundError at the user.
   Even if no automated check catches that intermediate state, a developer
   on the branch running `fincli --history` would hit a regression they
   can't easily diagnose. The configurator fix lands FIRST.

2. RESIST THE URGE TO BUNDLE.

   The tempting move is "do the fundainsight/ deletion + configurator fix
   in one commit, since they belong together conceptually." Don't. The
   commits are split deliberately so the configurator fix is a recoverable,
   bisectable, behavior-preserving point in history. If post-deletion
   something breaks and the fix turns out to need refinement, you want a
   commit that pins the fix in isolation.

3. THE Q3 = OPTION B REWRITE IS NOT OPTIONAL OR PARTIAL.

   When you reach commit 6, do not surgically edit ARCHITECTURE.md to
   delete the fundainsight section and call it done. The user explicitly
   chose Option B (full rewrite) because the reading experience matters:
   a future contributor opening ARCHITECTURE.md should see a coherent
   single-mode CLI architecture, not a two-mode doc with one mode
   excised. Rewriting is more work than redacting; redaction is the
   trap. Take the rewrite.

4. THE BANNER IS LITERALLY ONE LINE.

   The harness rollout spec banner (§6.5) is exactly one Markdown line
   under the existing title heading. Not a section. Not a header. One
   blockquote line. Body of the spec is HISTORICAL RECORD and stays.
   Resist any urge to "clean up" Phase-2 wording in the body — that
   would falsify the historical record.

5. FEEDBACK-LOG.md IS APPEND-ONLY.

   This is the most likely place an implementer accidentally violates the
   contract. Don't reformat existing entries. Don't fix typos. Don't
   reorder. Append exactly one new dated entry at the bottom (or wherever
   the file's existing convention places new entries — read the file
   first, match the convention, follow it).

6. THE TWO FOLLOW-UP SPECS ARE STUBS, INTENTIONALLY.

   docs/refactoring/cli-entry-point-spec.md and
   docs/refactoring/history-path-config-spec.md are deliberately
   stub-quality at creation time. The user has marked these as deferred,
   not as ready-to-action. Each spec should:
     - State the goal in a paragraph.
     - List the affected files.
     - List the open questions / decisions to make later.
     - Be skeleton-shaped per docs/refactoring/README.md naming convention.
   Do NOT do the design for these specs. Their detailed design is the
   user's choice when they're ready to action them.

7. THE GREP CHECK IS THE MOST DURABLE TEST WE HAVE.

   Without a test suite, the `grep -r "fundainsight"` and `grep -r
   "yahooquery"` checks are the strongest residual-detection mechanism in
   the verification checklist. Run them yourself before declaring done.
   The expected answers:
     - "fundainsight" — exactly ONE match: the banner line on the harness
                        rollout spec.
     - "yahooquery"   — ZERO matches.
   If you get unexpected matches, do not paper over them; surface back.

8. THE RUFF EXTEND-EXCLUDE PRUNE IS REAL CLEANUP.

   The wisdom_fruit, shared, example, src, benchmarks entries in
   [tool.ruff] extend-exclude exist BECAUSE those folders existed.
   Once they're deleted, the entries become noise pointing at non-existent
   paths. Ruff doesn't error on them, but they confuse future readers.
   Prune them in commit 1 alongside the deletion.

9. .GITIGNORE AUDIT — LOOK FOR FUNDAINSIGHT-SPECIFIC PATTERNS.

   Likely candidates: `fundainsight/local_history/` (entry that pinned
   the fundainsight history path). Verify `fincli/local_history/` is
   covered by an existing gitignore entry; if not, add it. Don't
   add fundainsight/local_history if absent — the directory is being
   deleted, the gitignore entry would be gibberish.

10. PRE-REFACTOR MYPY BASELINE.

    TESTING.md cites ~105 mypy errors across the current codebase as
    Phase 1 baseline. Capture the actual baseline before commit 1 by
    running `mypy fincli fundainsight core config logger` on the current
    branch. After commit 3 (when fundainsight/ is gone), run
    `mypy fincli core config logger` and confirm same-or-fewer errors.
    "Same-or-fewer" allows for the ~zero error contribution of
    fundainsight to disappear; this is expected.

11. CLEAN-VENV INSTALL IS NOT OPTIONAL.

    The `pip install -e ".[dev]"` from a clean venv is the only way to
    catch the project rename actually taking effect. Existing dev
    environments may have `finscrape` cached in egg-info; reinstalling
    on top doesn't always recover cleanly. Clean venv test is the
    canonical fix.

12. ASK BEFORE IMPROVISING.

    The spec is the formalization of approved decisions, not a
    rendezvous. If you find a real contradiction or a gap, surface it
    back to ARCH or HUMAN. Don't quietly improvise — that's how decisions
    get made in shadow.

Lessons Learned (mirror of spec §13)
------------------------------------------------------------------------------

The v1.2 validation cycle completed with VERIFIER returning VERIFIED,
REVIEWER returning APPROVE_WITH_NITS, and QA returning PASS — but only
because QA elected Path B (deferred the live interactive happy-path to the
user). On the user's first live run, an IndexError surfaced in
fincli/cli/cli_stock_screener.py that had been latent in master since
before this branch existed. The validation cycle's grep-and-import-based
smoke checks could not have caught it because the bug was in a
runtime-interactive code path with no automated coverage.

Two takeaways for future refactors:

  1. A "Behavior" acceptance criterion that requires a live network
     round-trip should not be deferred to the user without the user
     explicitly opting in. When QA reports Path B, the orchestrator should
     treat that as a pending acceptance check, not a closed one — and the
     user should be invited to validate before any "ready to merge" claim,
     not after.

  2. Refactor specs that say "no behavior change in the kept code" should
     still require a 30-second smoke of the kept code's user-facing
     surface. A refactor can pass every grep, lint, and type check while
     leaving a years-old user-facing bug unmasked. The §7.2 Amendment-A
     clean-venv check (added in v1.1) caught import-time failures; an
     analogous "interactive smoke" check would have caught this. Captured
     as a forward-looking item for Phase 2 of the harness rollout.
